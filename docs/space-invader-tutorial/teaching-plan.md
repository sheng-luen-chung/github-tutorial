# 講師備課:教學設計理念與帶課節奏

> 這份是給**教學者**看的設計說明;學生實作請看 [README.md](README.md) 與各階段講義。

## 核心設計理念:一個功能 = 一個分支 = 一個 PR

對剛上大一的學生,git 最難的不是指令,而是**「為什麼要這樣做」**。所以我們把每個遊戲改良都包裝成一次完整的 GitHub 協作循環,讓學生三次反覆操作同一套流程,肌肉記憶就建立起來了:

```
建立 Issue → 開 branch → 寫程式 → commit → push → 開 PR → merge → 切回 main → pull
```

遊戲功能由淺入深,git 概念也由淺入深,兩條線並行。**重點是 git 的熟練度,遊戲只是有趣的載體。**

## 三小時時間軸

| 時間 | 階段 | 遊戲功能 | Git / GitHub 學習重點 |
|------|------|----------|----------------------|
| 0:00–0:30 | 暖身 | 跑起現有遊戲、讀懂主迴圈 | 環境設定、`clone`、`status`、`log`、main 分支概念 |
| 0:30–1:15 | **階段一** | 玩家 3 條命 + 顯示 | branch → commit → push → **第一個 PR** → merge |
| 1:15–2:00 | **階段二** | 關卡系統(清空進下一關、敵人加速) | 再做一次完整循環、用 **Issue** 追蹤、寫好的 commit message |
| 2:00–2:50 | **階段三** | 防護罩 / 爆炸特效 | **故意製造 merge conflict** 並解決、`.gitignore` |
| 2:50–3:00 | 收尾 | 玩成果 | 回顧 commit 歷史、PR 紀錄、GitHub 貢獻圖 |

## 各階段設計用意

### 階段一:玩家生命值 —— 學「完整跑一次流程」
- **遊戲改動(小):** 單次被擊中即結束,改成 3 條命並顯示。約 10 行。
- **為什麼選它當第一個:** 程式碼簡單到不會卡住,讓學生把全部注意力放在 git 流程上。第一次成功 merge 自己的 PR 的成就感,是整堂課的鉤子。

### 階段二:關卡系統 —— 學「重複循環 + Issue + 好的 commit」
- **遊戲改動(中):** 敵人清空後重生一波、速度 +1、關卡數 +1。約 20 行。
- **為什麼選它:** 第二次跑流程已較順,可加入協作習慣 —— 先開 Issue 描述任務,commit 用 `Closes #1` 自動關閉 issue,並示範拆成多個小 commit。

### 階段三:防護罩 + 爆炸特效 —— 學「解決衝突」
- **遊戲改動(大):** 加入可被打掉的防護罩(bunkers)與擊落爆炸特效。
- **本堂高潮 —— 刻意安排 merge conflict:** 讓學生在分支裡改某一行(分數顏色),同時在 main 上也改同一行,merge 時就會衝突。帶他看懂 `<<<<<<<` `=======` `>>>>>>>` 三段標記,手動解決後 commit。順帶教 `.gitignore`(排除 `__pycache__/`)。
- 衝突解決是新生最害怕、卻最常遇到的,在受控環境體驗過一次,以後就不慌了。

## 給初學者的教學要點

1. **指令一律附「白話翻譯」**:`commit`=拍存檔快照、`push`=上傳雲端、`PR`=請求把我的修改併進去。
2. **每階段留 5 分鐘 buffer**:高中生打字慢、會 typo。
3. **永遠不在 main 直接寫程式**:從第一分鐘就養成開分支的習慣(階段三製造衝突是唯一的教學特例)。
4. **commit 訊息用中文沒關係**:重點是描述清楚「做了什麼」。
5. **遊戲跑不動時**(常見:GUI 卡住),Ctrl+C 或關視窗即可,別讓 debug 吃掉 git 教學時間。
