# 講師備課:教學設計理念與帶課節奏

> 這份是給**教學者**看的設計說明;學生實作請看 [README.md](README.md) 與各階段講義。
> 課程形式:**家教 / 兩堂制**,每堂 = 3 小時講解 + 2 小時回家自練。

## 核心設計理念:一個功能 = 一個分支 = 一個 PR

對剛上大一的學生,git 最難的不是指令,而是**「為什麼要這樣做」**。所以我們把每個遊戲改良都包裝成一次完整的 GitHub 協作循環,讓學生反覆操作同一套流程,肌肉記憶就建立起來了:

```
先拉、開枝、改存推、開 PR 合、再拉、剪枝
= checkout main → pull → checkout -b → 改 → add → commit → push
  → 開 PR → Merge → checkout main → pull → branch -d
```

完整版見 [workflow.md](workflow.md)。遊戲功能由淺入深,git 概念也由淺入深,兩條線並行。**重點是 git 的熟練度,遊戲只是有趣的載體。**

## 為什麼從 3 小時改成兩堂 × (3+2)?

原本想用一堂 3 小時塞完三個改良,實測太趕:高中生打字慢、git 觀念要時間消化、衝突處理更需要從容演練。改成兩堂後:

- 每個觀念都有時間「講 → 帶做 → 回家自己再做一次」
- 衝突處理(最難)獨立放第二堂,不被趕進度
- 兩份回家作業讓學生**獨立**跑流程,真正內化

## 各階段設計用意

### 階段一:玩家生命值 —— 學「完整跑一次流程」
- **遊戲改動(小):** 單次被擊中即結束,改成 3 條命並顯示。約 10 行。
- **為什麼第一個:** 程式碼簡單到不會卡住,讓學生把全部注意力放在 git 流程上。第一次成功 merge 自己的 PR 的成就感,是整堂課的鉤子。

### 階段二:關卡系統 —— 學「重複循環 + Issue + 好的 commit」
- **遊戲改動(中):** 敵人清空後重生一波、速度 +1、關卡數 +1。約 20 行。
- **為什麼:** 第二次跑流程已較順,可加入協作習慣 —— 先開 Issue,commit 用 `Closes #1` 自動關閉,並示範拆成多個小 commit。

### 階段三:防護罩 + 爆炸特效 —— 學「解決衝突」
- **遊戲改動(大):** 加入可被打掉的防護罩與擊落爆炸特效。
- **本堂高潮 —— 刻意安排 merge conflict:** 讓學生在分支與 main 改到同一行,merge 時衝突。帶他看懂 `<<<<<<<` `=======` `>>>>>>>`,手動解決後 commit。順帶教 `.gitignore`。
- 衝突解決是新生最害怕、卻最常遇到的,在受控環境體驗過一次,以後就不慌了。

## 給初學者的教學要點

1. **指令一律附「白話翻譯」**:`commit`=拍存檔快照、`push`=上傳雲端、`PR`=請求把我的修改併進去。
2. **每段留 buffer**:高中生打字慢、會 typo。寧可少講一點,也要讓他親手做完。
3. **永遠不在 main 直接寫程式**:從第一分鐘養成開分支的習慣(製造衝突是唯一教學特例,且要說明這是特例)。
4. **commit 訊息用中文沒關係**:重點是描述清楚「做了什麼」。
5. **遊戲跑不動時**(常見:GUI 卡住),Ctrl+C 或關視窗即可,別讓 debug 吃掉 git 教學時間。
6. **作業重點在流程不在功能**:批改時看分支/commit/PR 有沒有做對,功能對不對是其次。
