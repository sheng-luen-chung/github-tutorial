# Git 背景知識(先看這個,再開始動手)

寫給**完全沒碰過 git 的高中生**。先建立直覺,指令自然就好懂了。

## 1. 為什麼需要「版本控制」?

你一定做過這種事:

```
報告_最終版.docx
報告_最終版_真的最終.docx
報告_最終版_這次真的.docx
```

很亂,而且改壞了救不回來、兩個人一起改會互相覆蓋。**Git 就是專門解決這件事的工具** —— 它幫你:

- 記錄每一次修改(誰、何時、改了什麼),隨時能回到任何一個過去的版本
- 開「平行宇宙」(分支)做實驗,不影響正式版
- 多人一起改同一個專案不會打架

## 2. Git ≠ GitHub

這是新手最常搞混的:

| | 是什麼 | 類比 |
|---|---|---|
| **Git** | 裝在你電腦上的**工具**,負責記錄版本 | 像「存檔系統」 |
| **GitHub** | 網路上的**平台**,放 git 專案、給大家協作 | 像「遊戲存檔的雲端 + 社群」 |

可以只用 Git 不用 GitHub;但用了 GitHub 一定有 Git 在背後。

## 3. 四個核心名詞

```
Repository（repo,倉庫）
  └── 一個專案的資料夾,Git 在裡面記錄所有歷史

Commit（提交,存檔快照）
  └── 某個時間點的完整狀態 + 一句說明。像遊戲存檔點。

Branch（分支）
  └── 一條平行的開發線。在分支上亂改都不會動到正式版。

main（主分支）
  └── 預設的「正式版」分支,通常放穩定、可動的版本。
```

## 4. 本機 vs 遠端(最重要的觀念)

你的電腦上有一份(**本機 / local**),GitHub 上有一份(**遠端 / remote**)。它們是**兩份獨立的東西**,要靠指令同步:

```
   你的電腦 (local)                    GitHub (remote)
   ┌──────────────┐                  ┌──────────────┐
   │   commit     │  ── push ──→     │              │  上傳
   │              │  ←── pull ──     │              │  下載
   └──────────────┘                  └──────────────┘
```

- **push**(推):把本機的 commit 上傳到 GitHub
- **pull**(拉):把 GitHub 上的最新版下載回本機

> 🔑 記住:`commit` 只是存到**你自己電腦**。沒 `push`,GitHub 上看不到、別人也拿不到。

## 5. 為什麼要「開分支」而不是直接改?

直接在 main 上改,就像「直接在正式上線的網站上改程式」—— 改壞了大家都遭殃。

正確做法:開一條分支 → 在分支上慢慢改、實驗 → 確定 OK 了,才透過 **Pull Request** 請求併回 main。main 永遠保持乾淨可用。

## 6. Pull Request(PR)是什麼?

PR = 「**我這條分支改好了,請把它併進 main 好嗎?**」的一張申請單。

在 GitHub 上開 PR 後,可以:
- 看到這條分支到底改了哪些行(綠色新增、紅色刪除)
- 別人(或自己)審查、留言
- 確認沒問題後按 **Merge** 正式合併

公司、開源專案都靠 PR 協作。我們這套教程就是讓你把 PR 流程練到熟。

## 7. 衝突(conflict)不可怕

當**兩邊改了同一行**,Git 不知道該聽誰的,就會「衝突」,要你親手決定。它長這樣:

```
<<<<<<< HEAD
你這邊的版本
=======
另一邊的版本
>>>>>>> main
```

你只要:挑要留的、刪掉不要的、把 `<<<`/`===`/`>>>` 三行記號清乾淨,再 commit 就解決了。第二堂課會實際演練一次。

## 8. 這堂課你會一直用到的指令

```bash
git status                 # 我現在改了哪些檔案?(最常用,迷路就打它)
git log --oneline          # 看歷史:一行一個 commit
git branch                 # 我在哪條分支?(* 那條)
git checkout main          # 切到 main 分支
git checkout -b 名稱        # 建立並切到新分支
git add 檔名               # 把改動放進「準備存檔」區
git commit -m "說明"       # 拍一張存檔快照
git push                   # 上傳到 GitHub
git pull                   # 從 GitHub 下載最新版
```

> 卡住時,先打 `git status`,它幾乎都會告訴你下一步該做什麼。
